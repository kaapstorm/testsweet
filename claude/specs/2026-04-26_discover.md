# Design: `discover` function

## Scope

A pure discovery API: given an imported module, return the list of
`@test`-decorated functions reachable via that module's namespace.

Out of scope for this slice:

- Any CLI / `python -m assertions` entry point
- A test runner (calling the discovered functions, capturing exceptions,
  reporting pass/fail)
- `Test` base class discovery
- Parameterized tests (`test_params`)
- Discovery from a file path or package walk
- `assert_raises` / `assert_warns`

## Approach

Discovery is a plain function over `vars(module)`. No registries, no import
hooks, no caching. It reads the same `TEST_MARKER` attribute that the
`@test` decorator writes.

Imported names are included. Per the project's design principle ("stay
close to Python"), the user can leverage Python's import system to compose
test groups — `from tests.test_b import some_test` makes `some_test` part
of the importing module's namespace, and discovery follows.

## Public surface

```python
from assertions import discover

def discover(module: ModuleType) -> list[Callable]: ...
```

The caller is responsible for importing the module. The function does not
accept dotted strings, paths, or anything else. Future slices may layer
those on as separate functions.

## Behavior

For each `name, value` pair from `vars(module).items()` (in dict insertion
order, guaranteed by Python ≥3.7):

1. Skip values that are not callable.
2. Skip values where `getattr(value, TEST_MARKER, False) is not True`.
3. Append the value to the result list.

Properties:

- Returns a fresh list on each call.
- Does not mutate the module.
- Order is `vars()` iteration order — definition order for locally defined
  `@test` functions, import order for imported ones, whichever they
  appear first if the same name is rebound.
- Includes `@test` functions imported into the module's namespace.

## Module layout

| File | Action | Purpose |
|------|--------|---------|
| `src/assertions/_discover.py` | create | Implement `discover` |
| `src/assertions/__init__.py`  | modify | Re-export `discover` |

`_discover.py` is separate from `_markers.py` because they have distinct
responsibilities: `_markers.py` writes the marker; `_discover.py` reads
it. Future variants of discovery (from package, from path) will live
alongside `discover` in `_discover.py`.

## Tests

`tests/test_discover.py` (using `unittest`), covering:

1. Discovers a single locally-defined `@test` function.
2. Discovers multiple `@test` functions in definition order.
3. Skips undecorated functions in the same module.
4. Skips non-callable attributes that happen to carry the marker
   (defensive — confirms the `callable(...)` guard).
5. Includes a `@test` function imported via `from ... import ...`.
6. Order with mixed local and imported `@test` functions matches the
   module's `vars()` iteration order.
7. Returns an empty list for a module with no `@test`-decorated members.
8. Each call returns a fresh list — mutating one result does not affect
   the next.

Test fixtures: small Python modules under `tests/fixtures/` imported via
`importlib.import_module` inside each test. Fixtures are kept minimal and
named for the behavior they exercise.

## Acceptance

- `from assertions import discover` succeeds.
- All eight tests above pass under `uv run python -m unittest`.
- The marker tests from the previous slice (`tests/test_markers.py`)
  continue to pass.
